Backend & API Development (Python)
[ ] Setup:
[ ] Set up FastAPI or Flask project structure (create app, models, routers/services).
[ ] Configure Python environment, install required libraries (fastapi/flask, pydantic, supabase, httpx, etc.).
[ ] Authentication Middleware:
[ ] Implement middleware or dependency that validates Supabase JWT on protected routes.
[ ] Return 401 Unauthorized if token is missing or invalid.
[ ] API Endpoints:
[ ] Profile:
[ ] GET /profile – return authenticated user’s profile data.
[ ] PATCH /profile – allow user to update name, risk tolerance, etc.; validate input.
[ ] Portfolio:
[ ] GET /portfolio – return current holdings and their current values for the user.
[ ] POST /portfolio/transaction – record a buy or sell transaction (symbol, shares, price, type).
[ ] Transactions:
[ ] GET /transactions – return list of past transactions for the user.
[ ] POST /transactions – (if separate from portfolio endpoint) allow manual entry of transactions.
[ ] Recommendations:
[ ] GET /recommendations – compute or fetch and return personalized recommendations for the user.
[ ] Education Content:
[ ] GET /education – return list of available educational content items.
[ ] POST /education – (protected/admin) add new content item.
[ ] Chatbot:
[ ] POST /chat – accept a user question, call the LLM, and return AI-generated answer.
[ ] Market Data:
[ ] GET /market-data?symbol=XYZ – fetch and return real-time stock price and basic info for given symbol.
[ ] Supabase Integration:
[ ] Connect to Supabase database (using the Supabase Python client or direct PostgreSQL connection).
[ ] Implement CRUD operations for each table (users, portfolios, transactions, etc.) within endpoints.
[ ] Market Data Integration:
[ ] Choose and configure a stock data API (Alpha Vantage, IEX Cloud, Yahoo Finance).
[ ] Write service code to fetch current stock prices and handle API keys securely.
[ ] If streaming data is needed, implement WebSocket or Server-Sent Events (SSE) to push updates.
[ ] Implement caching of market data (in-memory or Redis) to respect API rate limits.
[ ] Recommendation Engine Logic:
[ ] Fetch user’s profile (risk level, goals) and portfolio data.
[ ] Apply algorithm or model to generate a list of recommended assets or asset classes.
[ ] (Optional) Integrate a simple ML model or rules engine for recommendations.
[ ] Store generated recommendations and rationale in the database (with user_id).
[ ] LLM (AI) Integration:

[ ] In /chat endpoint, send user message to LLM with the appropriate prompt and system instructions.
[ ] Parse and format the model’s response to return to the client.
[ ] Input Validation & Error Handling:

[ ] Return proper HTTP status codes for errors (400 Bad Request for validation, 401 Unauthorized, 500 Internal Server Error).
[ ] Handle edge cases (user not found, symbol not found, external API timeout) gracefully and log errors.

[ ] Sanitize inputs to prevent SQL injection (use prepared statements/ORM).

[ ] Testing (Backend):

Data & Database Schema
[ ] Database Tables:
[ ] users: id (uuid, primary key), email, name, risk_tolerance, created_at.
[ ] portfolios: id, user_id (fk to users), symbol, quantity, avg_cost, created_at.
[ ] transactions: id, user_id (fk), symbol, type (buy/sell), shares, price, timestamp.
[ ] recommendations: id, user_id (fk), recommendation_text, rationale, date_generated.
[ ] education_content: id, title, category, url, description.
[ ] chat_history: id, user_id (fk), messages (JSON or separate table for messages), timestamp.
[ ] Use Supabase dashboard or SQL migrations to create tables and fields.
[ ] Set up foreign key relationships and indexes (e.g., index portfolios(user_id), transactions(user_id)).
[ ] Configure Supabase Row Level Security (RLS) policies:
[ ] Allow each user to SELECT/INSERT/UPDATE only on rows where user_id = auth.uid.
[ ] (Optional) Create database triggers/functions (e.g., auto-update portfolio value after a transaction).
[ ] Seed database with test data (dummy users, sample portfolios and transactions) for development.
[ ] Enable database backups or point-in-time recovery (PITR) for production.
